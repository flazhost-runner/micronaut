# `io.micronaut.inject` Class Map

This module is the runtime contract layer for Micronaut dependency injection. Most classes in `io.micronaut.inject` are small interfaces that generated bytecode implements; the heavier runtime bases that implement those contracts live mostly in `io.micronaut.context`.

Use this file when changing `inject/src/main/java/io/micronaut/inject/**` to keep the generated-code model, bean lookup flow, and qualifier rules aligned.

## Runtime Shape

The normal bean lifecycle is:

1. Annotation processing writes a lightweight `BeanDefinitionReference` and a concrete `BeanDefinition`.
2. `BeanContext` discovers references without loading user bean classes, checks `BeanContextConditional`, qualifiers, indexes, and replacement rules, then calls `BeanDefinitionReference.load(...)`.
3. A loaded `BeanDefinition` describes the bean type, scope, constructor, injection points, executable methods, type arguments, metadata, and lifecycle hooks.
4. If it is an `InstantiatableBeanDefinition`, the context creates the instance with a `BeanResolutionContext`.
5. Generated definitions resolve constructor, method, field, `@Value`, `@Property`, provider, collection, stream, map, and registration dependencies through helper methods on `io.micronaut.context.AbstractInitializableBeanDefinition`.
6. If implemented, `ValidatedBeanDefinition`, `InjectableBeanDefinition`, `InitializingBeanDefinition`, and `DisposableBeanDefinition` add argument validation, member injection, `@PostConstruct`, and `@PreDestroy` behavior.
7. `ExecutableMethod` and `ExecutionHandle` expose reflection-free method calls for `@Executable`, AOP, event listeners, bean introspection methods, and processor startup work.

Generated classes are part of the API surface even when the supporting type is `@Internal`. Avoid changing method signatures, constructor shapes, visibility, or default method semantics unless the generated writers and compatibility impact are handled together.

## Core Bean Model

`BeanType<T>` is the common candidate abstraction. Both `BeanDefinitionReference<T>` and `BeanDefinition<T>` use it to expose the bean class, annotation metadata, exposed types, `@Primary`, optional bean name, candidate checks, and method-processing hint.

`QualifiedBeanType<T>` extends `BeanType<T>` for anything that can expose a declared `Qualifier<T>`. It resolves qualifier annotations from target annotation metadata and is the bridge between static metadata and the runtime `io.micronaut.context.Qualifier` API.

`BeanDefinitionReference<T>` is the soft-loading handle. It must answer whether the bean type is present, whether the bean is enabled, singleton, parallel, indexed, configuration-properties, proxy target, and how to load the real `BeanDefinition<T>`.

`BeanDefinition<T>` is the full description used after loading. It extends `QualifiedBeanType`, `Named`, and `ArgumentCoercible`, and supplies scope, constructor, required components, injection points, lifecycle methods, executable methods, type arguments, default implementation, replacement metadata, and debug descriptions.

`BeanContextConditional` is implemented by references, definitions, and configurations to decide whether something is enabled for a `BeanContext` and optional `BeanResolutionContext`.

`BeanConfiguration` groups definitions by package and condition. `ConditionalBeanConfiguration` is the runtime record returned by `BeanConfiguration.of(...)` and `disabled(...)`; it deliberately rejects custom configurations for internal `io.micronaut.*` packages.

`BeanIdentifier` is the stable instance key used by the context. `DefaultBeanIdentifier` is its internal string-backed implementation.

`ArgumentBeanType<T>` adapts a core `Argument<T>` into a `BeanType<T>`, mainly so qualifier and candidate logic can operate on an argument-like requested type.

## Bean Construction And Lifecycle

`InstantiatableBeanDefinition<T>` marks definitions that can construct instances. It creates a `DefaultBeanResolutionContext` for convenience when only a `BeanContext` is supplied.

`ParametrizedInstantiatableBeanDefinition<T>` is for definitions that need runtime-supplied `@Parameter` values. It finds required constructor arguments, validates/converts supplied values, and blocks no-argument instantiation.

`InjectableBeanDefinition<T>` is for existing instances that need generated member injection after construction.

`InitializingBeanDefinition<T>` invokes generated `@PostConstruct` hooks after dependency injection.

`DisposableBeanDefinition<T>` invokes generated `@PreDestroy` hooks during shutdown.

`ValidatedBeanDefinition<T>` delegates constructor/member/bean validation to `BeanDefinitionValidator` from the active context.

`ReplacesDefinition<T>` represents `@Replaces` matching logic returned by `BeanDefinition.getReplacesDefinition()`.

`DelegatingBeanDefinition<T>` decorates another definition by forwarding the full `BeanDefinition` surface. Use it when behavior must wrap a definition without copying its metadata model.

## Injection Points

`InjectionPoint<T>` is the base description of a place where the declaring bean needs a dependency. It exposes the declaring `BeanDefinition` and declared qualifier, and centralizes the `@Inject(required = false)` check.

`CallableInjectionPoint<T>` adds an `Argument<?>[]` list for constructors and methods.

`ConstructorInjectionPoint<T>` describes the constructor or factory method/field used to create a bean.

`MethodInjectionPoint<B, T>` describes injected methods and lifecycle methods. It records the method name, arguments, declaring type, and whether it is post-construct or pre-destroy.

`FieldInjectionPoint<B, T>` describes injected fields. Its `getField()` method is deprecated because resolving the `Field` initializes reflection metadata.

`ArgumentInjectionPoint<B, T>` describes an argument inside an outer callable injection point and also acts as `ArgumentCoercible<T>`.

Generated `AbstractInitializableBeanDefinition` turns compact generated references into these injection point objects lazily and configures them with the `Environment` when needed.

## Executable Methods

`MethodReference<T, R>` is metadata for a method: declaring type, name, target `Method`, arguments, return type, and annotation metadata.

`ExecutableMethod<T, R>` adds reflection-free invocation and source-like descriptions on top of `MethodReference`.

`ExecutableMethodsDefinition<T>` is a generated collection of executable methods. `BeanDefinition` delegates method lookup and startup method processing to it.

`DelegatingExecutableMethod<T, R>` wraps an executable method and forwards all method metadata and invocation.

`BeanDefinitionMethodReference<T, R>` associates an executable method with the owning `BeanDefinition`. `DefaultBeanDefinitionMethodReference` is the package-private delegating implementation created by `BeanDefinitionMethodReference.of(...)`.

`ExecutionHandle<T, R>` binds an executable method to a target instance and invokes without passing the instance as the first argument.

`MethodExecutionHandle<T, R>` is an `ExecutionHandle` that also exposes the underlying `ExecutableMethod`.

`UnsafeExecutionHandle<T, R>` is the internal variant that can invoke without argument validation.

## AOP And Proxies

`AdvisedBeanType<T>` is implemented by advised references and definitions to expose the intercepted target type for around or introduction advice.

`ProxyBeanDefinition<T>` marks a bean definition as an AOP proxy and exposes the target definition type and target bean type.

`io.micronaut.inject.proxy.InterceptedBean` marks generated proxy classes from the inject side.

`io.micronaut.inject.proxy.InterceptedBeanProxy<T>` marks proxies that expose the intercepted target and optionally whether the target is already cached. It also implements the internal `Qualified<T>` hook so generated proxies can receive a bean qualifier.

AOP proxy modes to keep distinct:

| Mode | Target ownership | Generated marker | Main behavior |
| --- | --- | --- | --- |
| Default around proxy | The proxy is the bean instance. For concrete classes it subclasses the target and calls `super` through generated bridge methods when proceeding. | `Intercepted` | Fast path for `@Around(proxyTarget = false)`. Intercepted methods run against `this`; no separate target bean is resolved. |
| Proxy-target around proxy | The proxy and target bean are separate. The proxy bean definition points back to the target definition. | `ProxyBeanDefinition`, `InterceptedProxy` | Used for `@Around(proxyTarget = true)`, factory-produced around-advised beans, and interface cases that need a backing target. The proxy resolves the target through `BeanResolutionContext.getProxyTargetBean(...)`. |
| Lazy proxy-target proxy | Same as proxy-target, but target resolution is deferred until `interceptedTarget()` is needed. | `InterceptedProxy` | Stores a copied `BeanResolutionContext` and resolves the target on demand. With `cacheableLazyTarget`, the target is cached after the first lookup. |
| Hotswap proxy-target proxy | Same as proxy-target, but the target reference is mutable. | `HotSwappableInterceptedProxy` | Stores the target behind a read/write lock and exposes `swap(...)` to replace it. |
| Introduction proxy | There is no concrete target implementation for introduced methods. | `Introduced` | Generated proxy implements the advised interface/type and delegates behavior to introduction interceptors. Abstract/interface methods require an introduction interceptor or adapter. |
| Runtime proxy | The generated bean definition asks a `RuntimeProxyCreator` to create the proxy object. | Runtime-specific proxy type | `DefaultRuntimeProxyDefinition` carries method metadata, interceptor data, target/proxy-target settings, and constructor values to the runtime proxy creator. |

`@Around(proxyTarget = false)` is the normal class-subclass mode. `@Around(proxyTarget = true)` creates a proxy bean around a separate target bean, which changes lifecycle and destruction behavior because the context must find and destroy the target definition separately. Factory methods with around advice are forced into proxy-target mode because the factory owns creation of the original produced object. For factory-produced classes with constructor arguments, `proxyTargetMode` controls whether proxy generation fails, warns, or proceeds.

Method interception flow:

1. The AOP writer records intercepted methods and interceptor binding metadata from class and method annotations.
2. The generated proxy stores `ExecutableMethod[] $proxyMethods` and `Interceptor[][] $interceptors`.
3. During proxy construction, `$proxyMethods` is filled from the generated executable-methods definition. In proxy-target mode, methods are looked up from the target `BeanDefinition` with `getRequiredMethod(...)`.
4. The constructor resolves the applicable interceptor array for each method through `InterceptorChain.resolveAroundInterceptors(...)` or `resolveIntroductionInterceptors(...)`, using the preselected interceptor bean registrations injected into the proxy constructor.
5. Each intercepted method override chooses the invocation target. Default around proxies pass `this`; proxy-target proxies pass the target field or call `interceptedTarget()` for lazy/hotswap cases.
6. The override creates a `MethodInterceptorChain` with the method-specific interceptors, target, executable method, and argument values, then returns `proceed()`.
7. In default around proxies, concrete methods get generated `$$access$$...` bridge methods so `proceed()` can call the original implementation via `super` without reflection. Introduction methods instead rely on introduction interceptors to provide the implementation.

`io.micronaut.aop.beandefinition.InterceptedParametrizedInstantiateBeanDefinition<T>` is the AOP-side bridge for parametrized bean definitions with `@AroundConstruct`. It extends `ParametrizedInstantiatableBeanDefinition<T>` and adapts the normal `Map<String, Object>` argument path into a constructor interceptor chain: generated code resolves ordered construction values with `resolveInstantiationValues(...)`, the default map-based `doInstantiate(...)` starts `ConstructorInterceptorChain.instantiate(...)`, and the chain eventually calls the generated object-array `doInstantiate(...)` through `InterceptedParametrizedBeanConstructor`.

`io.micronaut.aop.beandefinition.ProxyInterceptedParametrizedInstantiateBeanDefinition<T>` is the proxy-specific version of that bridge. Generated AOP proxy constructors append hidden parameters after the real constructor arguments: `BeanResolutionContext`, `BeanContext`, `Qualifier`, constructor interceptor registrations, and `InterceptorRegistry`. This interface extracts the pre-resolved constructor interceptors from the second-to-last constructor value and invokes `ConstructorInterceptorChain.instantiate(...)` with an additional proxy parameter count of `5`, so interceptors see only user constructor arguments while the final proxy constructor still receives its internal values.

## Qualifiers

Use `Qualifiers` as the factory entry point. It turns annotation metadata, names, type arguments, interceptor bindings, and composed qualifiers into `Qualifier<T>` instances.

Most concrete qualifiers extend `FilteringQualifier<T>`, which implements the common `reduce`, `filter`, and `doesQualify` loops over `BeanType` and `QualifiedBeanType`.

Important qualifier implementations:

| Class | Role |
| --- | --- |
| `AnyQualifier` | Matches every candidate and returns the first candidate for single lookup. |
| `NoneQualifier` | Matches candidates with no declared qualifier. |
| `PrimaryQualifier` | Matches `@Primary`, declared primary qualifiers, or primary-style candidate names. |
| `NameQualifier` | Matches `@Named`, `NameResolver`, simple names, and generated name-plus-type conventions. |
| `AnnotationQualifier` | Matches a runtime annotation instance. |
| `AnnotationMetadataQualifier` | Matches annotation metadata, including binding and non-binding member rules. |
| `AnnotationStereotypeQualifier` | Matches candidates by stereotype or repeatable stereotype container. |
| `RepeatableAnnotationQualifier` | Matches repeatable qualifier annotation values. |
| `CompositeQualifier` | Chains arbitrary qualifiers. |
| `FilteringCompositeQualifier` | Chains only `FilteringQualifier` instances for faster direct matching. |
| `TypeArgumentQualifier` | Matches candidate generic type arguments for a requested bean type. |
| `ClosestTypeArgumentQualifier` | Narrows generic candidates to the closest assignable type argument match. |
| `ExactTypeArgumentNameQualifier` | Matches a single generic argument by exact type name. |
| `MatchArgumentQualifier` | Matches full nested `Argument` type parameters, including covariance and contravariance modes. |
| `EachBeanQualifier` | Matches the origin bean definition for an `@EachBean` product. |
| `TypeAnnotationQualifier` | Implements `@Type`-style filtering by exposed/assignable classes. |
| `InterceptorBindingQualifier` | Matches interceptor beans against interceptor binding metadata. |
| `QualifierUtils` | Shared package-private name, annotation, and declared-qualifier matching helpers. |
| `Qualified` | Internal generated-code hook used to push a runtime qualifier into proxies/providers. |

When changing qualifier behavior, check both `BeanType` candidates and `QualifiedBeanType` candidates. Many lookups first reduce references before loading definitions, so relying only on loaded `BeanDefinition` behavior can break startup and indexing paths.

## Annotation Metadata

`io.micronaut.inject.annotation` supplies the compile-time metadata implementation used by generated definitions and executable methods.

`DefaultAnnotationMetadata` is the immutable runtime-oriented representation of declared annotations, all annotations, declared stereotypes, all stereotypes, and reverse stereotype indexes.

`MutableAnnotationMetadata` is the build-time mutable variant used by processors and helper definitions before metadata is emitted or adapted.

`AbstractAnnotationMetadata` supplies lazy synthesized annotation proxies and shared synthesis behavior.

`AnnotationMetadataSupport` owns core annotation defaults, repeatable containers, annotation type registration, and proxy construction.

`AnnotationMetadataReference` points to metadata by class name without unwrapping the target metadata. Generated code can use it as a stable metadata reference.

`AnnotationMetadataHierarchy` composes metadata layers, especially factory-produced beans and AOP method/type metadata, while preserving declared-vs-inherited lookup behavior.

`AbstractEnvironmentAnnotationMetadata`, `EnvironmentAnnotationMetadata`, `EnvironmentAnnotationValue`, `EnvironmentConvertibleValuesMap`, and `EnvironmentOptionalValuesMap` adapt annotation values to the active `Environment` and resolve placeholders lazily.

`EvaluatedAnnotationMetadata`, `EvaluatedAnnotationValue`, `EvaluatedConvertibleValuesMap`, and `MappingAnnotationMetadataDelegate` adapt metadata with expression values that require a configured `BeanContext`, owning `BeanDefinition`, or method arguments.

`AnnotationConvertersRegistrar` registers conversion from `AnnotationValue` objects to synthesized annotation instances.

`AnnotatedElementValidator` is the compile-time validation hook for annotated elements.

`AnnotationMetadataException` wraps metadata construction failures as `BeanContextException`.

## Bean Introspection Helpers

`io.micronaut.inject.beans` backs generated bean introspections, not dependency-injection definitions.

`AbstractInitializableBeanIntrospection` is the generated-introspection base. It stores property, method, constructor, and annotation metadata and dispatches property/method access without reflection.

`AbstractInitializableBeanIntrospectionAndReference` lets a generated introspection also serve as its own `BeanIntrospectionReference`.

`AbstractEnumBeanIntrospectionAndReference` adds enum constant metadata support.

`AbstractExecutableBeanMethod` adapts an introspection `BeanMethod` into an `ExecutableMethod`.

`KotlinExecutableMethodUtils` recognizes Kotlin suspend functions whose continuation returns `Unit`.

## Provider And Validation Helpers

`io.micronaut.inject.provider.AbstractProviderDefinition` is both a `BeanDefinitionReference` and an `InstantiatableBeanDefinition` for provider-like injection points. It inspects the current `BeanResolutionContext` segment to determine the requested generic bean and qualifier.

`JakartaProviderBeanDefinition` builds `jakarta.inject.Provider<T>` instances.

`JavaxProviderBeanDefinition` builds legacy `javax.inject.Provider<T>` instances when that type is present.

`BeanProviderDefinition` builds Micronaut `BeanProvider<T>` instances with lookup, iteration, streaming, qualifier composition, and definition access.

`ProviderTypeInformationProvider` marks provider types as wrapper types for `Argument` type analysis.

`io.micronaut.inject.validation.BeanDefinitionValidator` is the validation SPI used by `ValidatedBeanDefinition`.

`io.micronaut.inject.validation.RequiresValidation` is an internal compile-time marker that generated definitions use to know validation is required.

## Change Guidance

Keep these contracts in sync with generated code in `inject-java`, `inject-groovy`, `inject-kotlin`, and the writers under annotation-processing modules. A default method change here can affect already-compiled generated definitions.

Prefer adding new capability as default methods on interfaces only when old generated classes can keep working. If generated implementations must override a new method, update the relevant writers and tests together.

Avoid reflection in new runtime paths. Existing deprecated reflection accessors are compatibility escape hatches and should not become part of new behavior.

When a change touches bean selection, qualifiers, metadata, or lifecycle order, add focused tests that exercise lookup through `BeanContext`, not only direct unit calls on the helper class.
